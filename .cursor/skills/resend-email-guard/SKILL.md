# Resend Email Guard — Talok (SOTA March 2026)

## Architecture (3 layers, strict order)

```
Caller → email-service.ts → resend.service.ts → resend-config.ts → Resend SDK
```

1. **`lib/services/email-service.ts`** — Public facade. All app code imports `sendEmail` from here.
2. **`lib/emails/resend.service.ts`** — SDK layer. Retry (3×, exp. backoff), rate limit (5/min/dest), validation.
3. **`lib/services/resend-config.ts`** — Single source of truth for API key, from, replyTo. Priority: DB > ENV > default.

### Rules

- **NEVER** instantiate `new Resend()` outside `resend.service.ts`.
- **NEVER** call `fetch("https://api.resend.com/emails")` from Next.js code. Only allowed in Supabase Edge Functions (Deno runtime).
- **NEVER** import `resend.service.ts` directly from routes/components. Always go through `email-service.ts`.
- **ALWAYS** use `resolveResendRuntimeConfig()` for credentials. Never read `RESEND_API_KEY` directly.

## Templates — Single source of truth

- **Use** `lib/emails/templates.ts` (`emailTemplates.*`) for all new templates.
- **NEVER** write inline HTML email templates in route handlers, cron jobs, or Edge Functions.
- **NEVER** create templates in `email-service.ts:EMAIL_TEMPLATES` (legacy, frozen).
- If an Edge Function needs a template, delegate to `EMAIL_SERVICE_URL` (calls `/api/emails/send`).

## Resend SDK — Mandatory patterns

```typescript
const { data, error } = await resend.emails.send({
  from: resendConfig.fromAddress,
  to: recipients,
  subject: template.subject,
  html: template.html,
  replyTo: resendConfig.replyTo || undefined,
  tags: [{ name: 'type', value: 'password_reset' }],
  idempotencyKey: `password-reset/${userId}`,
});
if (error) throw new Error(error.message);
```

### SDK rules (official Resend docs, March 2026)

- **camelCase** for SDK params: `replyTo`, `scheduledAt`, `idempotencyKey` (NOT snake_case).
- **snake_case** only for raw REST API in Edge Functions: `reply_to`, `scheduled_at`.
- Handle `{ data, error }` response — SDK does NOT throw. Use `try/catch` only for network failures.
- Always `await` — `resend.emails.send()` returns a Promise.
- Import from `resend` — class is `Resend`.
- `html`, `text`, `react` are mutually exclusive with `template`.

### Checklist for every email send

- [ ] Uses `email-service.ts:sendEmail()` (not direct SDK)
- [ ] Template from `emailTemplates.*` (not inline HTML)
- [ ] Has `html` (required). `text` is auto-generated by Resend since Aug 2025; provide explicit `text` only for critical emails (password reset, OTP, legal notices)
- [ ] `from` address comes from `resend-config.ts` (never hardcoded)
- [ ] `idempotencyKey` for retryable/critical flows (format: `{type}/{entity-id}`, max 256 chars, expires 24h)
- [ ] `tags` for tracking (`type`, optionally `entity_id`)
- [ ] Error result is checked (`if (!result.success)`)
- [ ] `onboarding@resend.dev` is NEVER used in production `from`

## Batch sending (crons, bulk relances)

For sending to multiple recipients in crons/automation:

```typescript
const { data, errors } = await resend.batch.send(
  emails.map(e => ({
    from: config.fromAddress,
    to: e.recipient,
    subject: e.subject,
    html: e.html,
    idempotencyKey: `onboarding-24h/${e.profileId}`,
    tags: [{ name: 'type', value: 'onboarding_reminder' }],
  })),
  { batchValidation: 'permissive' }
);
```

- Max 100 emails per batch call.
- `batchValidation: 'permissive'` — sends valid emails even if some fail validation.
- `attachments` and `scheduledAt` are NOT supported in batch.
- Batch idempotency keys supported since June 2025.

## Environment variables

| Variable | Required | Purpose |
|----------|----------|---------|
| `RESEND_API_KEY` | Yes (prod) | Primary API key |
| `EMAIL_FROM` | Yes (prod) | Sender address (`Talok <noreply@talok.fr>`) |
| `EMAIL_REPLY_TO` | Recommended | Reply-to address |
| `NEXT_PUBLIC_APP_URL` | Yes | Base URL for links in emails |
| `EMAIL_FORCE_SEND` | Dev only | Set `true` to send real emails in dev |
| `CRON_SECRET` | Yes (prod) | Auth for cron endpoints |
| `INTERNAL_EMAIL_API_KEY` | Yes (prod) | Auth for `/api/emails/send` |
| `RESEND_WEBHOOK_SECRET` | Yes (prod) | Svix signing secret for webhook verification |

- `EMAIL_API_KEY` and `RESEND_FROM_EMAIL` are legacy aliases — prefer `RESEND_API_KEY` and `EMAIL_FROM`.
- `onboarding@resend.dev` is test-only. **NEVER** use in production `from`.

## Domain & deliverability

- Verify domain in Resend dashboard with SPF + DKIM + DMARC.
- Send from a subdomain (e.g. `mail.talok.fr`) to isolate reputation.
- Set up Resend webhooks for bounce/complaint/delivery monitoring (see below).
- **NEVER** send to addresses on the suppression list.

## Webhook monitoring (SOTA Jan 2026)

### Recommended: Webhooks Ingester

Resend provides an open-source [Webhooks Ingester](https://github.com/resend/resend-webhooks-ingester) (Jan 2026):
- Next.js app, 1-click deploy (Vercel/Railway/Render)
- Supports Supabase/PostgreSQL natively
- Svix signature verification built-in
- Stores all 17 event types, idempotent, with retry
- 30-day Resend data retention — ingester gives you permanent storage

### Alternative: custom endpoint

```typescript
import { Webhook } from 'svix';

export async function POST(req: Request) {
  const body = await req.text();
  const headers = Object.fromEntries(req.headers);
  const wh = new Webhook(process.env.RESEND_WEBHOOK_SECRET!);
  const event = wh.verify(body, headers);
  // Handle: email.bounced, email.complained, email.delivered, email.suppressed
}
```

### Events to handle

| Event | Action |
|-------|--------|
| `email.bounced` | Add to suppression list, notify admin |
| `email.complained` | Add to suppression list, disable marketing emails |
| `email.delivered` | Update delivery status in DB |
| `email.suppressed` | Log, never re-send to this address |
| `email.failed` | Log error, alert if pattern detected |
| `email.delivery_delayed` | Log warning |

## Suppression list (Jan 2026)

Resend exposes `suppressed` status in events since January 2026.
- Check suppression status BEFORE sending (bounce/complaint = auto-suppressed).
- Never force-send to a suppressed address.
- Monitor via Webhooks Ingester or `email.suppressed` event.

## Edge Functions (Deno)

Edge Functions (`supabase/functions/`) cannot import from `@/lib/`. They must:

1. Use `fetch("https://api.resend.com/emails")` with `RESEND_API_KEY` from `Deno.env`.
2. Duplicate `normalizeResendFromAddress()` locally (copy from `resend-config.ts`).
3. Use **snake_case** params (`reply_to`, not `replyTo`) — raw REST API.
4. Always set `reply_to`.
5. Prefer delegating to `EMAIL_SERVICE_URL` (calls `/api/emails/send`) when possible.

## Adding a new email flow — Workflow

1. **Template**: Add to `lib/emails/templates.ts` as a new function on `emailTemplates`.
2. **Send**: Call `sendEmail()` from `lib/services/email-service.ts` in your route/action.
3. **Tags**: Add `tags` with `type` value matching the template name.
4. **Idempotency**: Add `idempotencyKey` for any flow that could be retried.
5. **Test**: Use `EMAIL_FORCE_SEND=true` in dev, or Resend CLI (`resend emails send`).
6. **Text fallback**: Provide explicit `text` for critical transactional emails.

## Common mistakes to catch in review

| Mistake | Fix |
|---------|-----|
| `new Resend(process.env.RESEND_API_KEY)` in a route | Use `sendEmail` from `email-service.ts` |
| Inline HTML template in route handler | Move to `emailTemplates.*` in `templates.ts` |
| `from: "noreply@talok.fr"` hardcoded | Use `resend-config.ts` resolution |
| Missing error check after `sendEmail()` | Always check `result.success` |
| No `idempotencyKey` on payment/auth email | Add `{type}/{id}` key |
| `reply_to` (snake_case) in SDK call | Use `replyTo` (camelCase) for SDK |
| Testing with real emails in dev | Default is simulation; use `EMAIL_FORCE_SEND=true` only intentionally |
| Sending to suppressed address | Check suppression list before send |
| `try/catch` around `resend.emails.send()` for SDK errors | SDK returns `{ data, error }`, does not throw |
| Random UUID as idempotency key | Use deterministic `{type}/{entity-id}` format |

For detailed SDK parameters, idempotency patterns, and Talok file map, see [reference.md](reference.md).
